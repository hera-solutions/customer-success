## ERROR 1 — 403 AccessDeniedException on dashboard summary endpoints (HIGHEST priority)

**Production source on master:**
The three failing endpoints are called by these Vue components, which back the cards on `/da-management/dashboard`:
- [InfractionSummary.vue:165](src/views/Staff/components/SummaryCard/InfractionSummary.vue:165) — calls `API.post("heraPublicApi", "/dashboard/associate/infraction-summary", ...)`
- [IssueTrendsSummary.vue:143](src/views/Staff/components/SummaryCard/IssueTrendsSummary.vue:143) — calls `API.post("heraPublicApi", "/dashboard/associate/issue-trends-summary", ...)`
- [NetradyneAlertSummary.vue:138](src/views/Staff/components/SummaryCard/NetradyneAlertSummary.vue:138) — calls `API.post("heraPublicApi", "/dashboard/associate/netradyne-alert-summary", ...)`

The frontend code is correct — it uses Amplify's `API.post()`, which automatically signs requests with the logged-in user's Cognito credentials (the authenticated role).

**The actual problem is in the Amplify backend config**, file [cli-inputs.json](amplify/backend/api/heraPublicApi/cli-inputs.json): only the top-level path `/dashboard` is registered as a "private" (auth-required) route. The three sub-paths under `/dashboard/associate/*` are only defined inside the Express router inside the Lambda ([associateRouter.js:11](amplify/backend/function/dashboard/src/routers/associateRouter.js:11)). When Amplify generates the API Gateway, it builds an IAM policy that only allows the auth role to invoke `/dashboard` — not `/dashboard/associate/issue-trends-summary` etc. — so API Gateway 403s before the Express router ever runs.

**Already fixed in MQA or DQA?** **No.** All three branches have identical config — the three sub-paths are missing from `cli-inputs.json` on master, MQA, and DQA.

**Owner:** **MSP (AWS/infra).** This is an Amplify backend config + IAM policy issue, not frontend code.

**Suggested action:** Add the three sub-paths to [amplify/backend/api/heraPublicApi/cli-inputs.json](amplify/backend/api/heraPublicApi/cli-inputs.json) with `"permissions": { "setting": "private", "auth": ["create"] }`, then run `amplify push` to regenerate the API Gateway and the auth-role IAM policy. Open a new **HERA-** ticket for this — there is no in-flight fix to promote.

**Related hardening:** Audit every other Express router under `amplify/backend/function/*/src/routers/` — any nested sub-paths registered only inside Express (and not in `cli-inputs.json`) will have the same problem the moment someone calls them from the frontend.

## ERROR 2 — 401 "Token has expired" on AppSync calls

**Production source on master:**
- [src/main.js:108-118](src/main.js:108) — global `errorCatch()` matches the strings `"No current user"`, `"not authenticated"`, `"Not Authorized to access"` and immediately calls `router.push('/')` to bounce the user to login.
- [src/utilities/connectionHelper.js](src/utilities/connectionHelper.js) — the `safeFunction()` wrapper retries on **network errors only** (`"Network Error"`, `"Failed to fetch"`). It does **not** catch token-expiration errors, does **not** call `Auth.currentSession()` to refresh the token, and has no mutex to prevent multiple parallel refreshes.
- [amplify/backend/auth/hera4e362e32/cli-inputs.json](amplify/backend/auth/hera4e362e32/cli-inputs.json) — Cognito refresh-token lifetime is 30 days; the ID token uses Amplify's 1-hour default.

So: the user's refresh token is still valid for 30 days, but after the 1-hour ID token expires the very next API call throws an auth error, the global handler matches it, and the user gets kicked to login.

**Already fixed in MQA or DQA?** **YES — fixed in DQA only.**
- **Branch:** `origin/dqa`
- **Commit:** `ee343e8e5c` (also `62e72d26ad` merge commit, PR #17353)
- **Ticket:** **HERA-5609** — *"add token refresh mutex and stop auth error re-throwing"*
- **What it does:** Adds a single `refreshPromise` mutex in `connectionHelper.js` so concurrent expired-token requests share one `Auth.currentSession()` refresh; when refresh succeeds, the request retries silently. Also patches `NoteMixin.js` so auth errors show a "Session Expired" toast and stop propagating to the global handler instead of being re-thrown to it.

The fix is **not in MQA** and **not in master**.

**Owner:** **Core engineering (frontend).**

**Suggested action:** Promote **HERA-5609** from DQA → MQA → master. The fix already exists, has test coverage in DQA, and the code is well-scoped to two files.

**Related hardening:** Add a proactive refresh loop (e.g., refresh every ~50 minutes for a 1-hour token) so idle users never even see a 401. Sweep the rest of the code for places that swallow auth errors with a `router.push('/')` or `Auth.signOut()` without first trying to refresh.

## ERROR 3 — 403 InvalidSignatureException / clock skew (LOW priority)

**Production source on master:**
- [src/utilities/clockSync.js](src/utilities/clockSync.js) — calls a public `/time-sync-check` endpoint and compares server time to `Date.now()`.
- [src/views/Auth/Login.vue:38](src/views/Auth/Login.vue:38) — clock check runs on the login screen.
- [src/layouts/DefaultLayout.vue:384](src/layouts/DefaultLayout.vue:384) — clock check runs in `mounted()` on the default layout.

There is **no custom AWS signing code** — Amplify's SDK uses `Date.now()` under the hood for the SigV4 timestamp. The app does not set `AWS.config.systemClockOffset`, so a drifted client clock produces signatures the server rejects as "expired."

The clock check fires *after* login and *after* the layout mounts, but by that point dashboard API calls have already gone out — so the first wave of calls on a clock-drifted machine still fails.

**Already fixed in MQA or DQA?** **No** — same code on all three branches.

**Owner:** **Core engineering (frontend).**

**Suggested action:** Open a new HERA- ticket to (1) run the clock-sync check in `App.vue` *before* any authenticated route renders, and (2) compute a `systemClockOffset` from the server response and apply it to the Amplify/AWS SDK so signatures use server time instead of local time.

**Related hardening:** Add a recurring background re-sync every ~15 minutes so machines that drift mid-session don't start failing later in the day.

## ERROR 5 — 504 Gateway Timeout on `/messenger` → `get-staffs-unread-messages`

**Production source on master:**
- Handler: [amplify/backend/function/messages/src/app.js:30-76](amplify/backend/function/messages/src/app.js:30) — the GET `/get-staffs-unread-messages` route.
- DDB logic: [amplify/backend/function/messages/src/helper/dynamodbHelper.js:1-142](amplify/backend/function/messages/src/helper/dynamodbHelper.js:1) — the function `getMessagesByGroupAndMessageType`.
- Lambda config: [amplify/backend/function/messages/messages-cloudformation-template.json](amplify/backend/function/messages/messages-cloudformation-template.json) — `Timeout: 29` seconds, no explicit `MemorySize` (defaults to 128 MB).

**Why it times out (plain English):**
1. The function first queries the Message table by group to discover all staff/device IDs — with **no `Limit`**, it pages through *every* message for the tenant.
2. Then it queries MessageReadStatus for the entire group, again unbounded.
3. Then it makes **one DynamoDB query per staff** in batches of 10. For a tenant with hundreds of staff, that's hundreds of queries.

Lambda timeout is 29 seconds. API Gateway has a **hard** 29-second timeout. There is zero buffer — any tenant whose data takes more than ~28 seconds blows up with a 504. With only 128 MB of memory, the function is also CPU-starved (Lambda CPU scales with memory).

**Already fixed in MQA or DQA?** **No.** The handler is byte-identical on master, MQA, and DQA. (DQA commit `5286640693` HERA-8473 upgrades to Node 22.x, which helps GC slightly but doesn't fix the N+1 pattern.)

**Owner:** **MSP (backend Lambda + DynamoDB).**

**Suggested action:** Open a new HERA- ticket. Quick win: bump Lambda memory to 512 MB+ and add `Limit` clauses to the two scan-style queries. Real fix: page the per-staff lookups (the frontend's `totalOnly` query param suggests pagination was intended) or batch them more aggressively (25-50 per round instead of 10).

**Related hardening:** Grep `amplify/backend/function/` for other handlers that do "discover IDs → loop one query per ID" — that pattern will time out for any large tenant.

**Plain English:** The literal axios error string is `timeout of {value}ms exceeded` — the `{value}` resolved to `0`, which means the request was given a timeout but the value evaluated to 0 (most likely an environment variable read like `Number(process.env.VUE_APP_HTTP_TIMEOUT)` where the env var was missing, producing `NaN` → coerced to 0 in a `||`/ternary chain, or set to `0` by a stale interceptor). The exact origin couldn't be confirmed from code alone — there is no single axios setup file with a clearly broken default.

**Already fixed in MQA or DQA?** **No.** Two related-but-different commits exist:
- HERA-8251 (`7c24146368`) — adds 8-second timeouts to auth init calls.
- HERA-5609 (`32aafc1649`) — token refresh handling (= the Error 2 fix).

Neither addresses this specific bug.

**Owner:** **Core engineering (frontend / HTTP config).**

**Suggested action:** Open a new HERA- ticket. Add an explicit timeout (e.g., `timeout: 30000`) on every `axios.put/post/get` site under `src/views/`, and replace any env-driven timeout with `parseInt(process.env.VUE_APP_HTTP_TIMEOUT, 10) || 30000` so a missing/empty env value falls back to 30 seconds.
